# Nokoroa Code Review（並列専門家レビュー）

5つの専門家エージェントが `main...HEAD` の全差分を並列でレビューし、結果を集計して報告する。

## 専門家エージェント一覧

| # | 専門家 | 観点 |
|---|--------|------|
| 1 | コード品質 | 可読性・命名・DRY・YAGNI・不要コード・不要なログやコメント |
| 2 | セキュリティ | JWT認証・バリデーション・SQLi/XSS・CORS・環境変数管理 |
| 3 | パフォーマンス | N+1クエリ・Prismaのinclude最適化・非同期処理・不要データ取得 |
| 4 | API設計 | NestJS RESTful設計・DTO・ガード・エラーハンドリング・Prismaスキーマ |
| 5 | フロントエンド | Next.js App Router・MUI・React hooks・状態管理・レスポンシブ・a11y |

## 手順

### Step 1: 差分を分析して呼ぶ専門家を決定

```bash
git diff main...HEAD --name-only
git diff main...HEAD --stat
git log main...HEAD --oneline
```

**常時起動:** `コード品質`

**変更内容に応じて追加:**

| 条件 | 追加する専門家 |
|------|-------------|
| `nokoroa-backend/` 配下の変更 | `API設計`, `パフォーマンス` |
| `nokoroa-frontend/` 配下の変更 | `フロントエンド` |
| 認証・JWT・ガード関連の変更 | `セキュリティ` |
| `prisma/schema.prisma` の変更 | `API設計`, `パフォーマンス` |

### Step 2: 選定した専門家エージェントを並列起動

選定したエージェントを同一メッセージで Agent tool に渡し、並列実行する。

各エージェントのプロンプトには以下を含める:
- `git diff main...HEAD` の全差分
- レビュー観点の説明
- **指摘ごとに優先度ラベル(P0/P1/P2)を必須付与**
- レポート形式: `[P0/P1/P2] ファイル:行番号 — 指摘内容 → 修正提案`

### 優先度の定義

| ラベル | 意味 | 例 | アクション |
|-------|------|----|----|
| **P0** | Critical = マージブロッカー | セキュリティ脆弱性、本番障害確実、データ破損リスク | **必ず修正** |
| **P1** | Warning = 要修正だが議論可 | パフォーマンス問題、設計上の懸念、テスト不足 | 修正推奨、判断で見送り可 |
| **P2** | Suggestion = 任意改善 | 命名改善、リファクタ提案、コメント追加 | 任意 |

### Step 3: 結果を集計して報告

### Step 4: 統合判断フェーズ

全レビュー結果を集計後、以下の3段階で総合判定する:

| 判定 | 条件 | 推奨アクション |
|------|------|-------------|
| **マージ可** | P0=0 かつ P1=0、または P1がユーザー判断で見送り済み | PR作成・マージ |
| **要修正** | P0≥1 または P1の修正が妥当 | 該当箇所を修正→再レビュー |
| **設計見直し必要** | 複数のP0が異なる専門家から出ている / 同じ層で広範な問題 | 実装方針から再検討 |

## 出力フォーマット

```
## コードレビュー結果

### 変更サマリー
- ブランチ: feature/xxx | 変更: Nファイル / Nコミット
- 概要: [1行で]

### 専門家レビュー 集計（優先度別）

| 専門家 | 判定 | P0 | P1 | P2 |
|--------|------|----|----|----|
| コード品質 | ✅ | 0 | 0 | 1 |
| セキュリティ | ❌ | 1 | 0 | 0 |
| API設計 | ⚠️ | 0 | 2 | 0 |
| **合計** |  | **1** | **2** | **1** |

### 統合判断

**判定: ❌ 要修正** （P0=1のため）

理由:
- セキュリティ専門家の P0 指摘あり
- 修正推奨順序: P0（セキュリティ）→ P1（API設計のN+1）→ P2

### 詳細レビュー結果（優先度順）

#### P0 — マージブロッカー
- [セキュリティ] `nokoroa-backend/src/auth/jwt.guard.ts:45` — JWTシークレットがハードコード → 環境変数化必須

#### P1 — 要修正だが議論可
- [API設計] `posts.service.ts:120` — N+1クエリ → `include` で一括取得
- [API設計] `posts.controller.ts:30` — DTO未使用 → 入力バリデーション追加

#### P2 — 任意改善
- [コード品質] `feed.tsx:12` — 命名改善（`d` → `posts`）

### 次のアクション
- **マージ可**: 全P0/P1解消、PR作成へ
- **要修正**: P0/P1 修正 → 再レビュー
- **設計見直し**: 計画から再検討（autopilotならPhase 2へ戻る）
```
